# 微信投诉分段截图采集工具设计

日期：2026-05-19

## 概要

将现有“右侧长图拼接”方案改为“右侧分段截图”方案。

新版目标：
- 左侧投诉列表继续自动识别、自动点击、自动翻页
- 右侧不再输出一张长图，而是从当前投诉底部开始向上逐张截图
- 每个投诉输出到一个独立文件夹
- 每张截图尽量裁掉与上一张重复的底部内容，减少重复
- 优先追求稳定、速度和人工核对便利性，不再追求长图一次成型

## 用户流程

1. 用户打开微信电脑端消费者投诉页面，并保持窗口位置和大小不变。
2. 用户在工具里框选：
   - 微信窗口
   - 左侧投诉列表区域
   - 右侧完整内容区域
3. 用户点击“预览当前投诉分段图”：
   - 工具对当前已打开的投诉执行一次分段截图
   - 输出一个单独文件夹，内含多张顺序图片
4. 用户确认效果后点击“开始批量采集”：
   - 工具自动识别左侧当前屏可见投诉项
   - 逐个点击投诉项
   - 为每个投诉创建一个文件夹并执行右侧分段截图
   - 当前屏结束后滚动左侧列表，继续处理新项

## 核心设计

### 1. 左侧列表保持自动批量逻辑

左侧逻辑保留当前总体思路：
- 对 `list_region` 截图
- OCR 提取文本框
- 按纵向位置聚类出每个投诉项
- 生成点击中心点
- 使用 `昵称 + 金额 + 预览文本` 作为去重指纹
- 当前屏处理完后滚动左侧列表
- 连续两页没有新项时结束批量

这部分不依赖长图方案，因此只做最小改动。

### 2. 右侧改为分段截图，不再拼接

每个投诉的采集过程改为：
- 打开投诉后，右侧天然停在底部
- 截取 `content_region`，保存为 `001.png`
- 对右侧内容区域执行“向上查看更多历史内容”的滚动
- 每滚动一次，再次截取 `content_region`
- 将新截图和上一张截图比对，识别底部重复区域
- 若存在明显重复，则裁掉新截图底部重复部分，再保存为下一张
- 重复执行，直到判定滚动到顶部或没有新增内容

输出顺序固定为从下往上：
- `001.png` 是最底部当前页
- 更大的编号表示更靠上的历史内容

### 3. 重复裁剪策略

新方案不再尝试把所有图片拼成一张，而是只解决相邻两张之间的重复内容。

推荐策略：
- 每次保留第一张完整截图
- 对后续截图，只分析“新截图底部”和“上一张顶部或整体相邻区”的垂直重叠关系
- 在允许的重叠范围内寻找最佳匹配高度
- 匹配成功时，裁掉新截图底部的重复区，只保留新增的上半部分
- 如果匹配失败，则保守保存整张，避免误裁

这样做的好处：
- 速度快于长图拼接
- 就算个别页误判，也只影响当前一张，不会毁掉整条记录
- 更适合人工逐张查看和补充判断

### 4. 到顶停止条件

当前投诉在以下情况停止：
- 连续两次滚动后，新截图相对上一张几乎没有新增内容
- 新截图与上一张几乎完全一致
- 本次裁剪后的新增内容高度低于最小阈值，并连续出现

停止后：
- 结束当前投诉文件夹
- 返回批量调度流程，继续左侧下一位用户

### 5. 输出结构

每个投诉一个文件夹，命名规则保持现有风格：
- `0001_昵称_金额`
- `0002_昵称_金额`

文件夹内容：
- `001.png`
- `002.png`
- `003.png`

批量运行目录下保留：
- 每个投诉对应文件夹
- `run_manifest.json`

`run_manifest.json` 记录：
- 投诉序号
- 昵称
- 金额
- 输出文件夹路径
- 生成图片数量
- 成功/失败状态
- 失败原因

## 界面调整

主界面只做必要文案调整，不改变整体布局：
- `预览当前投诉长图` 改为 `预览当前投诉分段图`
- `开始批量采集` 保持不变，但其实际行为改为：
  - 自动点击左侧用户
  - 为每个用户输出一个分段截图文件夹
- 右侧区域仍然只需要框选一个完整内容区域

参数层面：
- 保留右侧滚动步长
- 保留滚动等待时间
- 保留左侧翻页比例
- 原先偏向长图拼接的阈值参数降级为“相邻截图重复裁剪阈值”

## 模块改动

### `capture.py`

新增或重构：
- 采集当前投诉的分段截图序列
- 右侧滚动后获取下一张截图
- 判断与上一张的重叠高度
- 裁掉重复区后保存
- 返回当前投诉生成的图片列表与张数

保留：
- 屏幕区域截图
- 右侧滚动与左侧滚动
- UI 稳定等待

### `automation.py`

改动批量流程：
- 不再为每个投诉调用“生成长图”
- 改为调用“生成一个投诉的分段截图文件夹”
- 成功记录改为保存目录与图片数量

### `session.py`

保留原有文件夹命名思路，并扩展：
- 为单个投诉生成目录
- 为目录中的图片生成顺序文件名

### `gui.py`

界面只做轻量调整：
- 按钮文案更新
- 预览成功提示改为显示目录和图片张数
- 批量状态里增加“当前投诉已保存多少张”

### `stitching.py`

原长图拼接器从主流程退场。

处理方式：
- 可以暂时保留文件，但不再作为主路径依赖
- 新主流程使用更轻量的“相邻截图重复裁剪”逻辑

## 错误处理

- 某个投诉采集失败时，只记录该投诉失败并继续下一个
- 如果连续失败达到阈值，则自动暂停批量任务
- 若某一张截图无法稳定判定重复区，保守保存整张，不中断当前投诉
- 若当前投诉一张图都没采到，则记为失败

## 测试计划

### 自动化测试

- 配置测试
  - 旧配置仍可加载
  - 新输出结果结构能正确序列化到 manifest
- 右侧采集测试
  - 首张图完整保存
  - 后续截图能裁掉重复底部
  - 匹配失败时保守保存整张
  - 连续无新增内容时停止
- 批量流程测试
  - 一个投诉对应一个目录
  - 目录中文件按 `001.png` 递增
  - 单条失败不影响后续项
  - manifest 正确记录目录和图片数量

### 手工测试

- 真机预览当前投诉，确认输出多张顺序图
- 短聊天场景，确认不会再出现长图大块重复
- 长聊天场景，确认能稳定向上截取直到顶部
- 批量场景，确认左侧自动点击和翻页仍可继续工作

## 非目标

本轮不处理：
- 重新做复杂控件识别
- 聊天文本结构化导出
- 把多张图再次自动合成长图
- 让用户在采集过程中同时操作微信

## 风险与取舍

- 分段截图会产生更多文件，但这是有意取舍，换取稳定性和速度
- 个别截图之间仍可能保留少量重复内容，但这比长图整条错位更容易接受
- OCR 左侧识别仍受微信页面清晰度影响，但该风险与本次右侧方案切换无直接冲突

## 实施建议

建议按以下顺序实现：

1. 先把“当前投诉分段截图”做稳定
2. 再把批量流程接到新的分段截图接口
3. 最后调整 GUI 文案与 manifest 输出
